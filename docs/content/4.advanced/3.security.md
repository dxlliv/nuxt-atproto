## Client-side OAuth model

nuxt-atproto runs OAuth **entirely in the browser**. Access and refresh tokens are managed by `@atproto/oauth-client-browser` in client storage — not on your Nitro server.

Implications:

- Any XSS vulnerability in your app can expose session tokens. Follow standard Vue/Nuxt XSS hygiene (sanitize user HTML, avoid `v-html` with untrusted content, keep dependencies updated).
- There is no server-side session to validate in API routes — authenticated XRPC calls happen from the client via `Agent`.
- This module intentionally does **not** implement a BFF or server-side token exchange. If you need server-held secrets or server-side posting, you must build a separate backend.

## Redirect and metadata integrity

- Serve `client-metadata.json` over **HTTPS** in production.
- `redirect_uris` must list every origin users sign in from. Typos and `localhost` vs `127.0.0.1` mismatches are the most common auth failures — not crypto bugs.
- `client_id` must match the canonical URL of your metadata file.

## Why no server OAuth

ATProto's browser OAuth flow uses DPoP-bound tokens and browser storage designed for SPAs. Server-side OAuth would require different token handling, cookie sessions, and threat assumptions. This module targets Nuxt apps that call `@atproto/api` from the client after login.
