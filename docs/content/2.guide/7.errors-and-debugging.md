## Error contracts

The module uses different error strategies depending on the API surface:

| API | On failure |
|-----|------------|
| `signIn()` | Catches errors, logs to `console.error`, **does not throw** |
| `signInWithHandle()` | Same — silent to callers |
| `signOut()` | Logs revoke errors, still clears local session |
| `restore(did)` | **Throws** if restore fails (propagates from OAuth client) |
| `useAtprotoAgent('authenticated')` | **Throws** `'Not authenticated'` when no session |
| `useAtprotoAgent(...)` on server | **Throws** `'useAtprotoAgent is only available on the client'` |
| Plugin `client.init()` | Logs error, sets `status` to `'anonymous'` |

### Sign-in appears to do nothing

Because `signIn` and `signInWithHandle` swallow errors, check the browser console for messages like `Error during sign-in:`. Common causes:

- Invalid or unreachable `client-metadata.json`
- `redirect_uris` mismatch with your app origin
- Network failure during `authorize()` for handle-based sign-in

Wrap calls if you need UI feedback:

```ts
const { signIn } = useAtprotoAuth()

async function onSignIn() {
  try {
    await signIn()
    // redirect happens — code after may not run
  }
  catch {
    // only if you change the module or call $atproto.client directly
  }
}
```

For user-visible errors before redirect, validate config at build time and surface OAuth metadata issues in [Troubleshooting](/advanced/troubleshooting).

### `Not authenticated` from agent

Call `useAtprotoAgent('authenticated')` only when `useAtprotoSession().isLogged` is true and `status !== 'initializing'`. Guard in `watch` or session hooks, not during SSR.

## Browser DevTools

### Console

Enable `atproto.debug: true` in `nuxt.config.ts` for OAuth init and sign-out logs. Session hooks fire regardless of `debug`.

### Network tab

During sign-in, look for:

- Fetch of your `client-metadata.json` (must return 200)
- Redirect to PDS OAuth endpoints
- Callback to your `redirect_uris` origin

### Application tab

`@atproto/oauth-client-browser` persists sessions in browser storage (IndexedDB / localStorage depending on version). If sessions behave unexpectedly, inspect stored entries — but treat them as opaque; use module APIs to manage sessions.

## Agent and XRPC errors

Errors from `@atproto/api` (rate limits, auth, validation) propagate normally from agent methods. Handle them in your `try/catch` around `getTimeline`, `post`, etc. The module does not wrap XRPC responses.

For ATProto API usage beyond authentication, see the [`@atproto/api` documentation](https://www.npmjs.com/package/@atproto/api).
